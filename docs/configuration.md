# Configuration of the OCP Controller

This document describes runtime configuration for the controller. The source of
truth is ProjectConfig in api/config/v2alpha2/projectconfig_types.go, and the
default values are in config/default/config.yaml.

## Multi-file configuration

The --config flag can be passed multiple times. Files are merged in order:

1. The first file is the base.
2. Each later file is an overlay.
3. Fields omitted in overlays are kept from earlier files.

A common setup is to keep non-secrets in a ConfigMap file and sensitive values
in a Secret file.

Example startup flags:

--config=/etc/styra-controller/config.yaml
--config=/etc/styra-controller-secrets/config-secrets.yaml

Example secret overlay:

apiVersion: config.bankdata.dk/v2alpha2
kind: ProjectConfig
opaControlPlaneConfig:
  token: my-ocp-token

## Top-level ProjectConfig fields

- controllerClass
- deletionProtectionDefault
- disableCRDWebhooks
- logLevel
- leaderElection
- opa
- systemPrefix
- systemSuffix
- opaControlPlaneConfig

## OPA Control Plane configuration

opaControlPlaneConfig supports:

- address
- token
- gitCredentials
- bundleObjectStorage
- defaultRequirements
- systemDatasourceChanged
- libraryDatasourceChanged

Notes:

- bundleObjectStorage currently supports S3 configuration only.
- This controller no longer handles direct MinIO/S3 credential provisioning;
  OCP should be configured through OCP-side secret references in the configured
  object storage settings.

## OPA runtime defaults

The opa section controls default OPA runtime config generated by the
controller. It includes decision logs, metrics, bundle persistence, bundle
server, and decision API reporting options.

## Observability

### Logging

Controller logs are written to stdout. Configure log verbosity with logLevel.
Values above 0 should normally only be used for debugging.

### Metrics

The controller exposes standard Go and controller-runtime metrics, plus:

- controller_system_status_ready: number of System resources in ready state.

## Multiple controller instances

When running multiple controller instances in the same cluster:

- Use controllerClass so each controller reconciles only matching resources.
- Only one instance should have CRD webhooks enabled.

ControllerClass matching uses the label key styra-controller/class.

## Delete protection

deletionProtectionDefault controls default delete protection behavior on
resources when the resource-specific setting is not provided.

## Leader election

Configure leader election via:

- leaderElection.leaseDuration
- leaderElection.renewDeadline
- leaderElection.retryPeriod
